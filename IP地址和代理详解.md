### 适配器说明

| 适配器 | IP | 作用 |
|--------|-----|------|

> ⚠️ 以上全部是**私有 IP**（`192.168.x.x`），仅用于局域网通信，没有物理地理位置信息。

## 二、私网 IP 与公网 IP（NAT 转换）

私有 IP 无法直接在互联网上通信，需要通过路由器做 **NAT（网络地址转换）** 转为公网 IP。

### NAT 转换流程

```
① 笔记本发出请求（私网）
   源IP: 192.168.0.132:54321
   目标: baidu.com

② 路由器 NAT 转换
   源IP: 36.33.231.85:12345   ← 替换成公网 IP + 新端口
   目标: baidu.com
   记录映射表: 54321 ↔ 12345

③ 百度回复
   目标IP: 36.33.231.85:12345

④ 路由器查表还原
   查映射表: 12345 → 192.168.0.132:54321
   送回笔记本
```

| 关键概念 | 说明 |
|----------|------|
| **地址替换** | 路由器把私有源 IP 替换为自己的公网 IP |
| **端口映射** | 用不同端口区分内网不同设备 |
| **映射表** | 路由器内存中维护，自动匹配请求与响应 |
| **对外透明** | 外部服务器只看到公网 IP，完全不知道内网结构 |

**真实公网 IP：`36.33.231.85` → 合肥，安徽（中国联通）**

## 三、ICS 网络共享与开发板

笔记本开启 **ICS（Internet 连接共享）** 后，GEC6818 开发板以笔记本为默认网关接入互联网：

```
开发板(192.168.137.x)
       │ 网线
       ▼
笔记本 以太网(192.168.137.1)
       │
  ICS NAT 转发（直接绑定 WLAN 物理网卡）
       │
       ▼
  WLAN 网卡 → 路由器(192.168.0.1) → NAT → 公网
```

> ICS 本质是一个简易 NAT 路由器，在系统底层直接操作网卡转发数据包，**不经过应用层**。

## 四、ipinfo.io 是什么

[ipinfo.io](https://ipinfo.io) 是一个 IP 地理定位查询网站，本质和百度、谷歌一样，是一台互联网服务器。

```
invoke-restmethod ipinfo.io
         │
         ▼ DNS 解析
    ipinfo.io → 216.239.34.21
         │
         ▼ HTTP 请求（请求自带来源 IP）
    "当前访问者的 IP 是什么？"
         │
         ▼ 服务器响应（查 IP 归属数据库）
    { "ip": "185.36.192.157", "city": "Singapore", ... }
```

### 笔记本（开代理）查询结果

```
ip       : 185.36.192.157
city     : Singapore
org      : AS38136 Akari Networks
```

### GEC6818 开发板查询结果

```json
{
  "ip": "36.33.231.85",
  "city": "Hefei",
  "region": "Anhui",
  "country": "CN",
  "org": "AS140726 UNICOM AnHui province network"
}
```

### 为什么两边公网 IP 不同？

| 设备 | 流量路径 | 公网 IP |
|------|----------|---------|
| 笔记本 | 应用 → 代理客户端 → VPN 隧道 → 新加坡 | `185.36.192.157`（新加坡） |
| 开发板 | 网线 → ICS NAT → WLAN 网卡 → 路由器直连 | `36.33.231.85`（合肥） |

- 笔记本的系统代理是**应用层**代理，拦截浏览器/应用流量
- ICS 是**底层** NAT 路由器，直接操作物理网卡转发数据包
- **ICS 绑定了 WLAN 物理网卡作为出口**，不经过代理客户端，因此**代理对开发板无效**

## 五、代理模式数据流详解

### 不开代理（笔记本 + 开发板都走合肥）

```
笔记本(192.168.0.132) ──▶ 路由器(192.168.0.1) ──▶ NAT ──▶ 36.33.231.85（合肥）
开发板(192.168.137.x) ──▶ ICS ──▶ WLAN ──▶ 路由器 ──▶ 36.33.231.85（合肥）
```

### 开代理（笔记本走新加坡，开发板仍走合肥）

```
笔记本(192.168.0.132)
    → 代理客户端(127.0.0.1:7890) → 加密 → 路由器 → 新加坡代理 → 185.36.192.157（新加坡）

开发板(192.168.137.x)
    → ICS NAT → WLAN 网卡（绕过代理） → 路由器 → 36.33.231.85（合肥）
```

## 六、TUN 模式为什么对 ICS 无效

TUN 模式本质是创建一个虚拟网卡，系统所有流量默认走这个虚拟网卡。

```
开发板 ──网线──▶ 笔记本以太网(192.168.137.1)
                       │
                  ICS NAT 转发
                       │
              ┌────────┼────────┐
              ▼                 ▼
      TUN 虚拟网卡         WLAN 网卡
      (代理隧道)          (直连宽带)
              │                 │
              ▼                 ▼
          被路由表跳过      ICS 默认走这里 → 合肥
```

**原因**：ICS 的 NAT 在系统底层直接绑定了 **WLAN 物理网卡**作为出口，不经过路由表，绕过了 TUN 虚拟网卡。

---

## 七、解决方案：让开发板也走代理

在 **控制面板\网络和 Internet\网络连接** 中：

1. **取消** WLAN 的共享
   - 右键 WLAN → 属性 → 共享 → 取消"允许其他网络用户连接"

2. **启用** TUN 虚拟网卡的共享
   - 右键代理创建的虚拟网卡（如 Clash/v2rayN 的 TUN 网卡）
   - 属性 → 共享 → 勾选"允许其他网络用户连接"
   - 选择"以太网"

3. 验证
   - 开发板上运行 `wget -qO- ipinfo.io`，应该和笔记本一样显示新加坡 IP

```
设置后：
笔记本 ──▶ TUN 虚拟网卡 ──▶ 代理隧道 ──▶ 新加坡（185.36.192.157）
开发板 ──▶ ICS ──▶ TUN 虚拟网卡 ──▶ 代理隧道 ──▶ 新加坡（一致！）
```
